# SSQuant v0.4.4 更新说明

**发布版本：0.4.4**

本版本在回测统计、多品种资金、报告与示例策略等方面做了大量改进，并修复了数据服务「备用地址」与历史 K 线请求不一致的问题。若你从更早版本升级，建议通读下文「升级提示」。

---

## 一句话概览

| 方向 | 你会感受到的变化 |
|------|------------------|
| 回测结果更可信 | 加仓、减仓、反手等复杂成交的配对与盈亏统计已修正 |
| 多品种资金更直观 | 可按比例给每个数据源分配资金，报告里的初始资金不再「虚高」 |
| 报告更好用 | HTML 里能筛反手、K 线上反手标记更清晰 |
| 数据服务更一致 | 只配置备用服务器时，鉴权与下载 K 线会走同一套地址 |
| 示例与文档 | 新增多个验证/演示策略，账户类 API 的说明与真实行为一致 |

---

## 一、回测统计与结果（核心）

**以前的问题**：在「多次开仓再平仓」「只平一部分仓位」「反手（平多开空 / 平空开多）」等情况下，旧版用简单规则配对买卖，导致胜率、盈亏、交易次数等统计不准；权益曲线也算错了复合反手；滑点在部分平仓时算多了；报告里的「盈亏比」公式与常见定义不一致。

**现在**：用更符合实盘逻辑的方式跟踪持仓与均价；反手类动作会进入权益与滑点计算；盈亏比改为「总盈利 ÷ 总亏损绝对值」的常见口径；修复个别情况下汇总信息报错、以及日志里数据源编号显示错误等问题。

**对你意味着什么**：若策略涉及加仓、分批平仓或反手，升级后回测数字会更接近真实逻辑，建议对重要策略重新跑一遍回测对照。

---

## 二、多品种、多周期资金分配

**以前的问题**：多个品种共用一个总资金（例如 10 万）时，报告里可能把每个数据源都显示成「各 10 万」，加起来和总资金对不上。

**现在**：

- 支持 **`capital_ratio` 比例**：例如给 A 权重 6、B 权重 4，则资金按 60% / 40% 划分（比例可任意缩放，会自动归一化）。
- 也支持继续用 **`initial_capital`** 给某个数据源指定绝对金额。
- 同一品种不同周期可以设不同权重（例如焦炭 1 分钟和 5 分钟各占不同比例）。
- 启动回测时会在控制台打印每个数据源的权重、占比和对应金额，方便核对。

不填比例时仍按旧行为均分，不影响老策略。

## 七、数据服务：备用地址与历史 K 线（v0.4.4 重要修复）

**现象**：有时账号鉴权能通过，但回测或预加载历史 K 线失败；或你明明配置了 `fallback_servers`（备用服务器），却感觉「备用没生效」。

**原因**：鉴权会按「主地址 + 备用列表」依次尝试，而下载历史 K 线之前**只读主配置里的 `api_url`**，没有同样遍历备用地址。若你只把地址写在备用里、主地址留空，就会出现「能登录、拉不到线」的矛盾情况。

**现在**：历史 K 线与预加载请求会按与鉴权一致的顺序访问多个 HTTP 节点；某一节点连不上或没有有效数据时，会自动尝试下一个。配置说明也在 `ssquant/config/_server_config.py` 注释里写清楚了。

### 一、回测模块 BUG 修复

#### 1.1 交易配对逻辑重写（`backtest_results.py`）

**问题**：原有 `j % 2` 索引配对逻辑在加仓/建仓场景下完全错误，导致盈亏、胜率、交易次数统计全部失真。

**修复**：替换为**有状态的加权平均价跟踪**，正确处理以下交易动作：
- 开多 / 开空（建仓和加仓）
- 平多 / 平空（全部平仓和部分平仓）
- 平多开空 / 平空开多（反手复合动作）

#### 1.2 权益曲线复合动作支持（`backtest_results.py`）

**问题**：权益曲线计算完全忽略 `平多开空` 和 `平空开多` 两种复合动作。

**修复**：新增 `elif` 分支正确处理复合动作的持仓、现金、保证金和累计成本。

#### 1.3 滑点计算修正（`backtest_results.py`）

**问题**：滑点使用原始 `volume` 计算，部分平仓时多算滑点。

**修复**：将滑点计算移入每个动作分支，使用对应的实际成交量。

#### 1.4 盈亏比公式修正（`backtest_results.py`）

**问题**：`profit_factor` 实际计算的是赔率（`avg_win / avg_loss`），而非标准盈亏比。

**修复**：改为 `total_gross_profit / abs(total_gross_loss)`。

#### 1.5 `get_summary()` 崩溃修复（`backtest_results.py`）

**问题**：`calculate_performance()` 添加的 `'performance'` 键导致后续 `get_summary()` 崩溃。

**修复**：增加类型和键名校验 `if not isinstance(result, dict) or 'symbol' not in result: continue`。

#### 1.6 变量遮蔽修复（`backtest_results.py`）

**问题**：外层循环变量 `i`（数据源索引）被内层循环 `for i, date in enumerate(...)` 遮蔽，导致日志输出错误的数据源编号。

**修复**：外层循环变量重命名为 `ds_idx`。

### 二、多品种资金分配

#### 2.1 报告初始资金修正（`backtest_results.py`）

**问题**：多品种共享 100k 资金回测时，报告将每个数据源的初始资金都显示为 100k（4 个 DS 合计 400k），与实际的 100k 严重不符。

**修复**：根据分配模式自动计算每个 DS 的初始资金：
- 均分模式：`initial_capital / num_data_sources`
- 自定义模式：按品种或 DS 级别的配置值

#### 2.2 `capital_ratio` 比例权重分配（`unified_runner.py`）

**新功能**：支持两种资金分配方式：

| 方式 | 配置项 | 示例 |
|------|--------|------|
| 比例权重 | `'capital_ratio': 6` | A=6, B=4 → A 占 60%, B 占 40% |
| 绝对金额 | `'initial_capital': 60000` | 直接指定金额 |

特性：
- **按数据源独立分配**：同品种不同周期可设不同权重（如 j888 1m=4, j888 5m=1）
- **自动归一化**：权重总和不限制（6:4 和 60:40 效果相同）
- **控制台打印分配表**：回测启动时自动显示每个 DS 的权重、占比和金额
- **向后兼容**：不填则均分，不影响现有策略

```
[资金分配] 总资金 100,000 元，按 capital_ratio 分配：
  j888    1m  权重 4  →   40.0%  →      40,000 元
  j888    5m  权重 1  →   10.0%  →      10,000 元
  jm888   1m  权重 3  →   30.0%  →      30,000 元
  jm888   5m  权重 2  →   20.0%  →      20,000 元
```

#### 2.3 Per-DS 资金传递（`backtest_core.py`）

将 `unified_runner.py` 计算的每个 DS 精确资金额通过 `ds._allocated_capital` 传递到 `backtest_results.py`，确保报告使用准确的 per-DS 初始资金。

### 四、HTML 报告增强

#### 4.1 复合动作支持（`html_report.py`）

- 交易筛选下拉框新增 `平多开空` 和 `平空开多` 选项
- K 线图交易标记正确显示复合动作的买卖点

#### 4.2 反手标记增强（`html_report.py`）

**背景**：`reverse_pos` 在 `next_bar_open` 等模式下会拆成「平多 + 开空」或「平空 + 开多」两条成交记录，原先仅按单笔 `action == 平多开空` 无法与两腿拆分对齐；且与普通买卖共用三角标记，反手不够醒目。

**改动**：
- 同一时间（同一 K 线横轴类别）下相邻的 **平多 + 开空**、**平空 + 开多** 合并为反手标记，不再各画一个普通买卖三角。
- 单笔 `action` 为 **平多开空** / **平空开多** 的仍单独归入反手层。
- 图例新增专用散点：**平多开空**（橙色菱形）、**平空开多**（青色星形）；普通买入/卖出仍为绿/红三角。

### 五、海龟策略修正（`B_海龟交易策略.py`）

#### 5.1 出场信号不触发

**问题**：出场条件使用当前 K 线的通道值（`exit_lower.iloc[-1]`），而当前 K 线的最低价已包含在 rolling min 中，导致 `close < min(lows including close)` 数学上不可能成立。

**修复**：改用前一根 K 线的通道值 `exit_lower.iloc[-2]`。

#### 5.2 加仓条件始终为真

**问题**：`last_entry_price` 每根 K 线都基于 `current_price` 重新计算，导致 `current_price >= last_entry_price + 0.5*ATR` 始终成立。

**修复**：引入模块级 `_turtle_state` 字典持久化存储每次实际入场价格。

#### 5.3 头寸计算使用动态权益

**问题**：`account_size` 始终取固定的 `initial_capital`，账户盈亏后头寸规模不会调整。

**修复**：改用 `api.get_balance()` 获取实时权益，亏损时自动缩小头寸，盈利时适度放大。

### 七、新增策略

#### 7.1 加仓验证策略（`B_加仓策略.py`）

专门用于验证回测模块在加仓/部分平仓场景下的正确性。策略逻辑：
- MA 金叉入场做多，MA 死叉入场做空
- 持仓期间每 10 根 K 线自动加仓 1 手
- 产生「多次开仓 → 一次平仓」的交易序列

#### 7.2 减仓验证策略（`B_减仓策略.py`）

专门用于验证回测模块在**仅部分平仓（减仓）**场景下的统计正确性（与加仓策略互补：加仓侧重「多次开仓 → 一次平仓」，减仓侧重「一次开多手 → 分批平 1 手 → 反向信号全平」）。策略逻辑：
- MA 金叉/死叉时一次性开 `initial_lots` 手（默认 3 手），不做金字塔加仓
- 持多：RSI 超买且持仓大于 `min_lots` 时 `sell(volume=1)` 分批减仓；持空对称使用 `buycover(volume=1)`
- 反向均线交叉时平掉剩余全部仓位
- 可配 `reduce_interval`、`rsi_overbought` / `rsi_oversold` 等参数

#### 7.3 正反手策略（`B_正反手策略.py`）

演示 `api.reverse_pos()`：空仓时均线金叉/死叉普通开多、开空；持有多头遇死叉、持有空头遇金叉时反手换向（对应平多开空 / 平空开多）。可与 HTML 反手标记对照查看。

#### 7.4 正反手混合开平仓策略（`B_正反手混合开平仓策略.py`）

同一文件内混合演示 **多开/空开**（空仓均线开仓）、**多平/空平**（RSI 极端时 `sell` / `buycover` 全平）、**正反手**（均线反向交叉时 `reverse_pos`），并按优先级分支避免同一根 K 线重复下单。

> 说明：该示例最初文件名为 `B_混合开平仓策略.py`，已更名为 **`B_正反手混合开平仓策略.py`**，与「正反手策略」示例区分命名。

#### 7.5 示例策略注释更新

为 6 个多数据源示例策略统一添加 `capital_ratio` 和 `initial_capital` 的内联注释：

```python
{
    'symbol': 'rb888',
    'kline_period': '1m',
    'slippage_ticks': 1,
    # 'capital_ratio': 1,     # 资金权重（不填则均分，如 A=6,B=4 即 A 占 60%）
    # 'initial_capital': 1,   # 或直接指定金额（如 60000）
},
```

### 八、data_server 历史 K 线 REST 与 fallback_servers 对齐（0.4.4）

#### 8.1 问题

- **鉴权**（`auth_manager`）按「顶层 `api_url` + `fallback_servers[*].api_url`」顺序访问 `/api/auth/verify`。
- **历史 K 线**（`api_data_fetcher._try_data_server_api`、预加载 `historical_preloader._fetch_from_data_server`）此前**仅读取** `DATA_SERVER['api_url']`，未遍历 `fallback_servers`。
- 当用户仅注释顶层主地址、只在 `fallback_servers` 中配置 HTTP 时，会出现：**鉴权成功但回测拉取 K 线失败**（或误以为「备用服务器无效」）。

#### 8.2 修复

- 新增 **`get_ordered_data_server_api_bases()`**（`auth_manager.py`）：与鉴权、WS 端点顺序一致的 HTTP 基址列表；无配置时回退默认远端。
- **`_build_verify_urls()`** 改为基于上述基址拼接 `/api/auth/verify`，避免重复逻辑。
- **`_try_data_server_api`** 与 **`_fetch_from_data_server`** 对多个基址**依次请求** `/api/futures/history`，连接失败或无效响应时尝试下一节点。
- **`config/_server_config.py`** 注释说明：可与 `fallback_servers` 二选一或组合；仅配备选时 REST 与鉴权一致使用备选列表。
