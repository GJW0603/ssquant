## 2026-03-13 BUG修复与性能优化

| 文件 | 修改类型 |
|------|----------|
| `ssquant/backtest/live_trading_adapter.py` | BUG修复 + 性能优化 |
| `ssquant/pyctp/real_trading_client.py` | BUG修复 |
| `ssquant/data/ws_kline_client.py` | BUG修复 |

### 1. CTP 线程阻塞（性能优化）

**问题**：`_on_market_data` 在 CTP MD 线程上同步处理 tick，策略执行慢时阻塞后续 tick 接收。

**修复**：
- 新增 tick 队列 + 独立处理线程，CTP 回调只入队立即返回
- `_check_order_timeout` 节流为每秒最多检查一次
- `get_prev_trading_day` 结果按交易日缓存

### 2. 持仓查询重复累加

**问题**：平今转平昨后持仓查询不清理适配器缓存，`_on_position` 将新数据累加到旧数据上，导致持仓翻倍。

**修复**：
- `_on_position` 使用 `_position_seen_keys` 跟踪当前查询周期，每个 key 首次出现时覆盖旧数据而非累加
- `_on_position_complete` 清理 seen_keys 为下一次查询做准备

### 3. 强平未处理

**问题**：`_on_trade` 只处理了开仓/平仓/平今/平昨，交易所强制平仓时本地持仓不会更新。

**修复**：`offset_flag in ('1', '2')` 合并处理平仓和强平，`offset_map` 增加 `'2': '强平'`

### 4. 终态订单未清理

**问题**：`_on_order` 只在全部成交和全部撤单时移除 pending_orders，部分成交部分撤单的终态订单一直残留，导致反复尝试撤单。

**修复**：终态判断扩展为 `order_status in ['0', '2', '4', '5']`

### 5. 实盘缺少 is_ready()

**问题**：`_send_order` 调用 `self.is_ready()`，但 `RealTradingClient` 未定义（SIMNOW 有），实盘下单报 `AttributeError`。

**修复**：添加 `is_ready()` 和 `is_connected()` 方法，Spi 添加 `connected`/`logged_in` 标志

### 6. 断线后认证重试不取消

**问题**：`_retry_authenticate` 使用 `_was_connected`（永远为 True）检查连接状态，断线后认证重试不会被取消。

**修复**：改为检查 `self.connected`（断线时自动设为 False），增加 `self.logged_in` 检查

### 7. WebSocket 重连后鉴权丢失

**问题**：data_server 重启后 IP 鉴权记录清空，WebSocket 重连被 4001 拒绝，客户端不会自动重新鉴权，导致无限重连失败。

**修复**：
- `WSKlineClient` 新增 `on_auth_required` 回调，检测关闭码 4001 后标记需重新鉴权
- 每次重连前检查标志，触发 `reset_auth()` → `verify_auth()` → 重连成功

### 部署

替换以下文件：
- `ssquant/backtest/live_trading_adapter.py`
- `ssquant/pyctp/real_trading_client.py`
- `ssquant/data/ws_kline_client.py`

## 2026-03-26 实盘抗压增强

| 文件 | 修改类型 |
|------|----------|
| `ssquant/backtest/live_trading_adapter.py` | 实盘抗压增强 + 队列监控 |
| `ssquant/api/strategy_api.py` | 运行时状态接口增强 |
| `ssquant/config/trading_config.py` | 配置项新增 |

### 1. Tick 队列：有界抗压

**问题**：主队列无上限，行情活跃时持续积压。

**增强**：
- 新增 `tick_queue_maxsize` 配置，主队列改为有界队列
- 增加软水位和恢复水位机制
- 队列满时按品种缓存最新 Tick，队列回落后自动回灌
- 高压时压缩积压数据，只保留每个品种的最新 Tick

### 2. 运行时压力等级

让策略感知底层是否进入 Tick 积压状态，便于主动降级。

**统计项**：队列长度、高水位、overflow 缓冲区大小/累计次数、Tick 处理耗时、压缩次数

**压力等级**：`normal` → `busy` → `critical`

### 3. StrategyAPI 新增接口

- `get_runtime_stats()` — 获取运行时统计
- `get_runtime_pressure()` — 获取压力等级
- `is_runtime_under_pressure()` — 是否处于高压

### 4. Tick 队列调参建议

| 场景 | 建议值 |
|------|--------|
| 低频/少品种 | `5000 ~ 10000` |
| 中频/10~20 品种 | `15000 ~ 30000` |
| 高频/30+ 品种或夜盘活跃 | `30000 ~ 50000` |

### 5. 兼容性

- 策略函数签名不变（`initialize(api)` / `strategy(api)`）
- 回测引擎入口不变
- 现有 K 线驱动策略可直接使用

- `ssquant/config/trading_config.py`

## 2026-03-29 本地复权算法

| 文件 | 修改类型 |
|------|----------|
| `ssquant/data/local_adjust.py` | **新文件**：前复权/后复权算法 |
| `ssquant/config/trading_config.py` | 开启本地复权开关 |
| `ssquant/data/api_data_fetcher.py` | 复权提示信息适配 |
| `ssquant/data/historical_preloader.py` | 复权提示信息适配 |
| `ssquant/backtest/live_trading_adapter.py` | 复权提示信息适配 |

### 1. 复权算法

data_server 只存储不复权（raw）数据，复权计算在框架本地完成。

**实现**：
- 通过 `real_symbol` 列检测合约切换点
- 在切换点计算比例因子：`factor = 前一根close / 切换后第一根open`
- 后复权（`adjust_type='1'`）：最早数据保持原价，后续累积因子调整
- 前复权（`adjust_type='2'`）：最新数据保持原价，历史数据反向调整

**使用方式**：

```python
from ssquant.data import get_futures_data
df = get_futures_data('au888', '1D', adjust_type='1', limit=300)  # 后复权
df = get_futures_data('au888', '1D', adjust_type='2', limit=300)  # 前复权
```

### 2. 配置

`trading_config.py` 中 `ENABLE_REMOTE_ADJUST` 改为 `True`，数据获取时自动执行本地复权。

## 2026-03-31 自动移仓引擎

| 文件 | 修改类型 |
|------|----------|
| `ssquant/backtest/live_trading_adapter.py` | 旧合约报单匹配修复，策略前调用移仓引擎 |
| `ssquant/backtest/rollover_engine.py` | **新文件**：移仓状态机 |
| `ssquant/backtest/rollover_audit.py` | **新文件**：移仓复盘日志 |
| `ssquant/api/strategy_api.py` | 新增移仓状态查询接口 |
| `ssquant/config/trading_config.py` | 新增 `auto_roll_*` 配置项 |

### 1. 智能追单与旧合约修复

平旧合约时 CTP 回报的 `InstrumentID` 为旧合约，与当前 `ds.symbol`（新主力）不同。原逻辑只按新主力匹配，导致旧腿不入队、撤单合约错误、重发不触发。

**修复**：按「新主力 **或** 旧合约」统一匹配，撤单使用订单上的实际合约代码。

### 2. 自动移仓

仅 SIMNOW / 实盘生效，回测不支持。

**使用方式**：在 `get_config()` 中设置 `auto_roll_enabled=True`

**两种模式**：

| 模式 | 说明 |
|------|------|
| `simultaneous`（默认） | 同一次策略调用内平旧 + 开新，不等成交，速度更快 |
| `sequential` | 先平旧，旧腿平仓确认后再开新，更稳妥 |

**策略内查询**：`api.is_rollover_busy()` / `api.get_rollover_status()`

### 3. 复盘日志

默认开启（`auto_roll_log_enabled=True`），日志目录 `./live_data/rollover_logs/`，可选 `.jsonl` 格式。

### 部署

替换上述文件并同步 `trading_config.py`。

## 2026-04-02 回测资金约束修复

| 文件 | 修改类型 |
|------|----------|
| `ssquant/data/data_source.py` | 开仓资金校验、资金预占、拒单回传 |
| `ssquant/backtest/backtest_core.py` | 账户绑定、资金不足提前停止、拒单摘要 |
| `ssquant/backtest/backtest_logger.py` | 新增 `log_important()` |
| `ssquant/backtest/unified_runner.py` | 注入 `symbol_configs` 到策略参数 |

### 1. 开仓不受初始资金约束

**问题**：回测的开仓逻辑只改持仓、不看账户资金，策略可以无限开仓；待执行订单也没有做资金预占。

**修复**：
- 所有开仓统一按账户可用资金校验
- 待执行开仓单入队时预估资金占用，避免同一根 K 线里重复超额开仓
- 资金不足时自动裁剪手数；连 1 手都开不了则拒单

### 2. 策略读取的初始资金与回测配置不一致

**问题**：策略从 `strategy_params['symbol_configs']` 读取资金参数，但回测入口未注入实际配置，策略可能回退到默认值。

**修复**：回测启动前将实际生效的 `symbol_configs` 注入 `strategy_params`。

### 3. 资金不足后仍空跑整场回测

**问题**：首笔开仓被拒后，回测继续跑完全部时间区间，用户只看到"100% 完成"但 0 交易。

**修复**：
- 资金不足且无持仓、无待执行订单时，框架提前停止回测
- 仍有持仓或待执行订单时继续运行，保证闭环

### 4. 控制台看不到拒单原因

**问题**：资金不足等关键信息只写在日志文件里，终端里看不到。

**修复**：新增 `log_important()`，即使 `debug=False` 也会在控制台打印：
- 资金不足导致的开仓拒单
- 因资金不足提前停止
- 回测结束时的拒单统计摘要

### 部署

- `ssquant/backtest/backtest_core.py`
- `ssquant/backtest/backtest_logger.py`

## 2026-04-02 data_server 开盘假死修复

### 问题描述

在 `kline_source='data_server'` + `enable_tick_callback=True` 模式下，开盘时（如 09:00-09:03）由于第一根 K 线尚未从远程服务器推送，但 CTP 已经开始推送大量 TICK 数据（34 个品种同时推送），每个 TICK 都会触发一次完整的策略调用（含 `_strategy_lock` 加锁、`_rollover_engine` 遍历、`on_bar` 调用等），导致：

- TICK 队列持续积压、无法消费
- 系统表现为"假死"，策略无法正常执行
- 对于 3 分钟、5 分钟等较长周期策略尤为严重（第一根 K 线到达前所有 TICK 都会逐个触发策略）

### 修复方案

#### 1. 新增 tick 回调节流机制（核心修复）

**文件**：`ssquant/backtest/live_trading_adapter.py`

在 `_process_tick_data()` 方法中，当 `kline_source='data_server'` 且 `enable_tick_callback=True` 时，引入智能节流：

- **新 K 线到达时**：立即触发策略（通过 `threading.Event` 信号，由 `_on_ws_kline` 在收到完整 K 线时 `set()`）
- **无新 K 线时**：按 `tick_callback_interval`（默认 0.5 秒）间隔触发
- **队列积压时**（qsize > 500）：自动将间隔提升至 `max(tick_callback_interval, 2.0)` 秒
- **被跳过的 TICK**：清理 `_current_tick` / `_current_tick_symbol` 临时属性，避免脏数据

节流逻辑仅作用于 `data_server` 模式，`local` 模式完全不受影响。

#### 2. update_tick 性能优化

**文件**：`ssquant/backtest/live_trading_adapter.py` — `LiveDataSource.update_tick()`

在 `data_server` 模式下跳过两个最耗时的操作：

| 操作 | local 模式 | data_server 模式 |
|------|-----------|-----------------|
| `pd.to_datetime()` | 保留（K 线聚合需要） | **跳过** |
| `tick_data.copy()` | 保留（防止原始数据被篡改） | **跳过**（直接引用） |
| `tick_data['datetime']` | `pd.Timestamp` 对象 | 字符串（如 `"2026-04-02 09:00:00.500"`） |
| K 线聚合 | 执行 `_aggregate_kline()` | **跳过**（`return None`） |

日期时间的格式化（CTP TradingDay → 自然日转换、`datetime_str` 生成）在两种模式下均保留执行，确保 `tick['datetime']` 字段始终可用（用于下单价格参考等场景）。

#### 3. 新增配置参数 `tick_callback_interval`

**文件**：`ssquant/config/trading_config.py`

```python
'tick_callback_interval': 0.5,   # 秒，仅 data_server + tick回调 模式生效
```

| 值 | 行为 |
|----|------|
| `0.5`（默认） | 无新 K 线时每 500ms 触发一次策略 |
| `0` | 关闭节流，每个 TICK 都触发策略（适合需要逐 TICK 止盈止损的策略） |
| `1.0` / `2.0` | 更宽松的节流，适合低频策略 |

#### 4. 启动日志增强

**文件**：`ssquant/backtest/live_trading_adapter.py` — `__init__()`

当 `data_server` + `enable_tick_callback` 同时启用时，启动日志明确输出节流参数：

```
[实盘适配器] ✓ TICK流双驱动模式已启用（data_server节流: 新K线立即触发, 无K线时≤0.5s触发一次）
```

### 影响范围

| 场景 | 是否受影响 | 说明 |
|------|-----------|------|
| `data_server` + `enable_tick_callback=True` | **是** | 节流生效，解决假死问题 |
| `data_server` + `enable_tick_callback=False` | 否 | 无 TICK 回调，不涉及节流 |
| `local` + `enable_tick_callback=True` | 否 | 节流条件不匹配，逻辑不进入 |
| `local` + `enable_tick_callback=False` | 否 | 完全无影响 |

`update_tick()` 的优化同样仅作用于 `data_server` 模式，`local` 模式的 K 线聚合、`pd.to_datetime`、`tick_data.copy()` 逻辑完全保持原样。

### 部署

需要更新以下文件到服务器：

1. `ssquant/backtest/live_trading_adapter.py` — 核心修复（节流逻辑 + update_tick 优化）
2. `ssquant/config/trading_config.py` — 新增 `tick_callback_interval` 默认配置
3. `vip30/run.py` — 注释修正（可选）
